# Migrating to build-publish-novo 2.0

**Who this is for:** projects already on a `build-publish-novo` **1.x** release. If you are coming
from the old, non-`novo` plugin, start with the
[legacy → novo migration](./legacy-to-novo.md) first, then follow this guide.

2.0 reshapes two DSL areas — changelog **issue sources** and Jira **automation projects** — and
renames one CLI option. Each change is small and mechanical; the sections below show the exact
before/after. There are no functional regressions: everything the 1.x DSL did is still possible,
just expressed through the new containers.

---

## 1. Changelog: `issueNumberPattern` / `issueUrlPrefix` → `issueSources { }`

**What changed.** The foundation `changelog { common { } }` block no longer has the loose
`issueNumberPattern` and `issueUrlPrefix` properties. Issue tracking is now modelled as one or more
named **sources**, each pairing a `numberPattern` (the issue-key regex) with an optional `urlPrefix`
(the link base). This is what enables a single changelog to link keys from several trackers/hosts,
each to its own URL.

```kotlin
// OLD (1.x) — removed, no longer compiles
changelog {
    common {
        commitMessageKey.set("CHANGELOG")
        issueNumberPattern.set("TICKET-\\d+")
        issueUrlPrefix.set("https://jira.example.com/browse/")
    }
}
```

```kotlin
// NEW (2.0) — single source: use the issueSource(...) shorthand
changelog {
    common {
        commitMessageKey.set("CHANGELOG")
        issueSource("tracker") {
            numberPattern.set("TICKET-\\d+")
            urlPrefix.set("https://jira.example.com/browse/")
        }
    }
}
```

```kotlin
// NEW (2.0) — multiple sources: wrap them in issueSources { }
changelog {
    common {
        commitMessageKey.set("CHANGELOG")
        issueSources {
            issueSource("app") {
                numberPattern.set("APP-\\d+")
                urlPrefix.set("https://jira1.example.com/browse/")
            }
            issueSource("legacy") {
                numberPattern.set("LEG-\\d+")
                urlPrefix.set("https://jira2.example.com/browse/")
            }
        }
    }
}
```

The unnamed `issueSource { … }` shorthand also works when you only have one source and don't care
about its name.

**Behavior:** extraction (Jira/ClickUp automation) unions every source's `numberPattern`; link
rendering (Slack/Telegram) resolves each matched key against **its own** source's `urlPrefix`.

**To migrate:** move each old `issueNumberPattern`/`issueUrlPrefix` pair into one `issueSource`. If
you only had one, the single-source shorthand keeps it a one-liner change. See the
[Foundation plugin reference](../plugins/foundation.md).

## 2. Jira automation: automation-level properties → `projects { project() }`

**What changed.** The automation-level `projectKey`, `labelPattern`, `fixVersionPattern` and
`targetStatusName` properties are removed. A Jira automation rule now targets one or more
**projects**, each self-contained with its own key, optional Jira instance (`instanceName`) and
automation actions. Changelog issues are routed to a project by their issue-key prefix.

```kotlin
// OLD (1.x) — removed, no longer compiles
buildPublishJira {
    automation {
        common {
            projectKey.set("PROJ")
            labelPattern.set("build-%s")
            fixVersionPattern.set("v%s")
            targetStatusName.set("Ready for QA")
        }
    }
}
```

```kotlin
// NEW (2.0) — single project: use the project(...) shorthand
buildPublishJira {
    automation {
        common {
            project("main") {
                projectKey.set("PROJ")
                labelPattern.set("build-%s")
                fixVersionPattern.set("v%s")
                targetStatusName.set("Ready for QA")
            }
        }
    }
}
```

```kotlin
// NEW (2.0) — multiple projects, optionally across different Jira instances (see §3)
buildPublishJira {
    auth {
        common {
            instance("default") { /* primary instance: baseUrl + credentials */ }
            instance("legacy")  { /* second instance: baseUrl + credentials */ }
        }
    }
    automation {
        common {
            projects {
                project("app") {
                    projectKey.set("APP")
                    fixVersionPattern.set("v%s")
                }
                project("legacy") {
                    projectKey.set("LEG")
                    instanceName.set("legacy")   // route to the "legacy" instance
                    targetStatusName.set("Done")
                }
            }
        }
    }
}
```

The `project { … }` unnamed shorthand works for a single project. Unknown `instanceName`s and duplicate
project keys now fail fast at configuration time. See the
[Jira plugin reference](../plugins/jira.md).

**To migrate:** wrap your existing automation body in a single `project("main") { … }` (or the
unnamed `project { … }`). Only split into multiple projects / add `instanceName` if you actually target
more than one project or Jira instance.

## 3. Jira auth: named `instance(...)` instead of `buildVariant(...)`

**What changed.** A Jira `auth` block no longer holds a single base URL + credentials keyed by
build variant. It now holds one or more named **instances** (each its own base URL + credentials),
declared with `instance("name") { … }` inside `common { }` (or `buildVariant(...) { }`). This mirrors
how Telegram declares multiple `bot("name")`s and makes credentials a first-class, variant-independent
axis. A project selects an instance via `instanceName`; a project without `instanceName` uses the instance
named `default`, or the single instance when only one is declared.

```kotlin
// OLD (1.x) — credentials sat directly under common / buildVariant(...)
buildPublishJira {
    auth {
        common {
            baseUrl.set("https://primary.atlassian.net")
            credentials.username.set(providers.environmentVariable("JIRA_USER"))
            credentials.password.set(providers.environmentVariable("JIRA_TOKEN"))
        }
        buildVariant("legacy") {   // abused the variant axis to add a second credential set
            baseUrl.set("https://legacy.atlassian.net")
            credentials.username.set(providers.environmentVariable("LEG_USER"))
            credentials.password.set(providers.environmentVariable("LEG_TOKEN"))
        }
    }
}
```

```kotlin
// NEW (2.0) — named instances under common
buildPublishJira {
    auth {
        common {
            instance("default") {
                baseUrl.set("https://primary.atlassian.net")
                credentials.username.set(providers.environmentVariable("JIRA_USER"))
                credentials.password.set(providers.environmentVariable("JIRA_TOKEN"))
            }
            instance("legacy") {
                baseUrl.set("https://legacy.atlassian.net")
                credentials.username.set(providers.environmentVariable("LEG_USER"))
                credentials.password.set(providers.environmentVariable("LEG_TOKEN"))
            }
        }
    }
}
```

**To migrate:** wrap the body of each `common { }` / `buildVariant(...) { }` auth block in
`instance("default") { … }`, and turn any extra credential set you previously expressed as
`buildVariant("legacy") { … }` into a sibling `instance("legacy") { … }` under `common`. Project
`instanceName`s already reference these names, so no automation change is needed. The Jira standalone
tasks' `--instanceName` option selects an instance the same way.

## 4. Standalone CLI: `--issueNumberPattern` → repeatable `--issuePattern`

The Jira and ClickUp automation standalone tasks replace the single `--issueNumberPattern` option
with a repeatable `--issuePattern`, mirroring multi-source changelogs. Jira standalone tasks
(`addJiraFixVersion`, `addJiraLabel`, `transitionJiraIssue`) also gain an optional `--instanceName` to
select the instance.

```bash
# OLD (1.x)
./gradlew clickUpAutomationRelease --issueNumberPattern='APP-\d+'

# NEW (2.0) — one pattern
./gradlew clickUpAutomationRelease --issuePattern='APP-\d+'

# NEW (2.0) — multiple patterns (union)
./gradlew clickUpAutomationRelease --issuePattern='APP-\d+' --issuePattern='LEG-\d+'
```

**To migrate:** rename `--issueNumberPattern` to `--issuePattern` in any CI scripts; pass one
`--issuePattern` per pattern you need. See [standalone tasks](../standalone-tasks.md).

## Non-breaking improvements (no action required)

- **Groovy `Closure` overloads** across the DSL (`issueSources`/`issueSource`, `projects`/`project`,
  Telegram `bot`/`chat`/`destinationBot`, and every `plugin-sender` block) — Groovy scripts get
  proper delegation and IDE support automatically.
- **`build-publish-novo-core` is now an `api` dependency** of the plugins. Its public DSL types
  (strategies, `CollectionStrategy`, config base types) are available transitively — you can remove
  any manual `implementation("ru.kode.android:build-publish-novo-core:…")` you previously added just
  to reference them.
- **Jira behavior:** project keys are uppercased on entry; a project whose key prefix matches no
  changelog issue logs an info message and is skipped (instead of skipping silently).

## Migration checklist

- [ ] Replace `issueNumberPattern` + `issueUrlPrefix` with `issueSource("name") { numberPattern; urlPrefix }` (wrap in `issueSources { }` if more than one).
- [ ] Wrap Jira automation bodies in `project("name") { … }`; move `projectKey` / `labelPattern` / `fixVersionPattern` / `targetStatusName` inside.
- [ ] Wrap Jira `auth` credentials in `instance("default") { … }`; turn any `buildVariant("x")` credential set into a sibling `instance("x") { … }` under `common`.
- [ ] Add per-project `instanceName` only if you target more than one Jira instance.
- [ ] Rename `--issueNumberPattern` → `--issuePattern` in CI scripts (repeat per pattern).
- [ ] Remove any manual `build-publish-novo-core` dependency (now transitive via `api`).
- [ ] Bump the plugin/library version to `2.0.0`.
