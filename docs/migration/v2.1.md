# Migrating to build-publish-novo 2.1

**Who this is for:** projects already on **2.0**. If you are on a 1.x release, do the
[2.0 migration](./v2.md) first, then this guide.

2.1 adds **automatic changelog titles**: when a commit carries `CLOSES:`/`FIXES:` with an issue
number, the plugin fetches that issue's title from Jira (or ClickUp) and inserts it into the
changelog — so you no longer have to hand-copy the title into a `CHANGELOG:` line. Enabling it
required reshaping the Jira **projects** DSL into a shared registry. That reshape is the only
**breaking** change; everything else is additive and opt-in.

There are no functional regressions: everything 2.0 automation did is still possible, expressed
through the new two-level project selection.

---

## 1. Jira projects: self-contained automation projects → a shared registry (breaking)

**What changed.** In 2.0 each automation project was self-contained — it carried its own
`projectKey` and optional `instanceName`:

```kotlin
// OLD (2.0) — removed, no longer compiles
buildPublishJira {
    auth {
        common {
            instance("default") { baseUrl.set("https://jira.example.com"); /* creds */ }
            instance("legacy")  { baseUrl.set("https://legacy.example.com"); /* creds */ }
        }
    }
    automation {
        common {
            projects {
                project("app") {
                    projectKey.set("APP")
                    fixVersionPattern.set("%s")
                }
                project("legacy") {
                    projectKey.set("LEG")
                    instanceName.set("legacy")
                    targetStatusName.set("Done")
                }
            }
        }
    }
}
```

In 2.1 a project is **declared once** in a registry nested under its `auth` instance (the same way
Telegram chats are declared under a bot). Consumers — `automation` and the new `issueResolution` —
then **select** projects by name, two-level: an instance name plus its project names. `projectKey`
now lives **only** in the registry and must be globally unique.

```kotlin
// NEW (2.1)
buildPublishJira {
    auth {
        common {
            instance("default") {
                baseUrl.set("https://jira.example.com")
                credentials.username.set("service-account")
                credentials.password.set(providers.environmentVariable("JIRA_API_TOKEN"))
                project("app") { projectKey.set("APP") }   // registry entry (nested directly on the instance)
            }
            instance("legacy") {
                baseUrl.set("https://legacy.example.com")
                credentials.username.set("u"); credentials.password.set("p")
                project("legacy") { projectKey.set("LEG") } // registry entry
            }
        }
    }
    automation {
        common {
            // select an instance, then its projects; override patterns per project if needed
            targetInstance("default") {
                project("app") { fixVersionPattern.set("%s") }
            }
            targetInstance("legacy") {
                project("legacy") { targetStatusName.set("Done") }
            }
            // automation-level patterns still work as defaults for every selected project:
            // labelPattern.set("release"); fixVersionPattern.set("%s"); targetStatusName.set("Ready for QA")
        }
    }
}
```

**To migrate:**

1. Move every `projectKey` out of `automation { projects { … } }` and declare each project directly
   on its instance with `project("localName") { projectKey.set("KEY") }` inside the matching
   `auth { … instance("…") { … } }`. Pick a short local name per project (e.g. `app`, `legacy`).
   Make sure each `projectKey` is unique across all instances.
2. In `automation`, replace `projects { project("x") { projectKey; instanceName; patterns } }` with
   one `targetInstance("<instanceName>") { … }` block per instance. Inside it, select the registry
   projects with `projectNames("app", "other")`, or with `project("app") { …pattern overrides… }`
   when you need per-project pattern overrides (referencing the project by its **registry** name).
3. Delete every `projectKey` and `instanceName` from the automation blocks — they are resolved from
   the registry now.

Unknown instance names, unknown project names, and duplicate `projectKey`s fail fast at
configuration time.

## 2. New: auto-resolve `CLOSES:` / `FIXES:` to changelog titles (opt-in)

**What's new.** Previously, to get a titled changelog entry you wrote **two** lines per commit — a
team `CLOSES:`/`FIXES:` line the plugin ignored, and a hand-copied `CHANGELOG: [KEY] <title>` line.
2.1 lets the plugin fetch the title itself, so a single `CLOSES:`/`FIXES:` line is enough. Manual
`CHANGELOG:` lines still work and coexist (a commit carrying both forms for the same issue yields one
entry).

This is nothing to migrate — existing changelogs are unchanged until you opt in. To adopt it:

**a. Declare the commit markers** in `buildPublishFoundation`:

```kotlin
buildPublishFoundation {
    changelog {
        common {
            commitMessageKey.set("CHANGELOG")     // manual entries still work
            issueReferences {                      // NEW — auto-resolved markers
                issueReference("closes") { key.set("CLOSES") }   // numberPattern defaults to (\d+|[A-Z]+-\d+)
                issueReference("fixes")  { key.set("FIXES") }
            }
        }
    }
}
```

**b. Enable a provider to resolve them.** For Jira, add an `issueResolution { }` block selecting the
instance/project the plugin should read titles from:

```kotlin
buildPublishJira {
    // … auth { } registry as above …
    issueResolution {
        common {
            fromInstance("default") { projectNames("app") }
        }
    }
}
```

With exactly one project selected, **bare numbers** resolve against it (`CLOSES: 3458` → `APP-3458`).
Select more than one project only if your commits use **prefixed** keys (`CLOSES: APP-3458`,
`FIXES: LEG-42`) — bare numbers become ambiguous across multiple projects and are left unresolved.

ClickUp mirrors this with its own `issueResolution { common { } }` block; each
provider handles only the references it recognises, so both can be declared together.

**c. (Optional) tune rendering.** Both strategies live in
`buildPublishFoundation { changelog { common { } } }` and never fail the build:

```kotlin
changelog {
    common {
        // how a RESOLVED reference renders (default: KeyAndTitleResolvedStrategy → "[APP-3458] Title")
        resolvedIssueStrategy { TitleOnlyResolvedStrategy }        // or KeyOnlyResolvedStrategy

        // how an UNRESOLVED reference renders (default: ChangelogLineOrKeyUnresolvedStrategy)
        unresolvedIssueStrategy { KeyOnlyUnresolvedStrategy }      // or SkipUnresolvedStrategy /
                                                                   // FallbackTextUnresolvedStrategy("(no description)")
    }
}
```

See the [foundation](../plugins/foundation.md) and [Jira](../plugins/jira.md) plugin docs for the
full reference.
